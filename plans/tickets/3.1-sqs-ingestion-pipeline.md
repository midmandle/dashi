# 3.1 — Introduce SQS into the ingestion pipeline

Currently the Ingestion API writes events directly to DynamoDB. This ticket inserts SQS between the Ingestion API and a new Event Processor service to decouple ingestion from storage and absorb write spikes as agent count scales. The change is transparent to agents — the published event format is unchanged.

**Expected outcome:** Events are enqueued to SQS by the Ingestion API and written to DynamoDB by the Event Processor. Ingestion throughput is no longer constrained by DynamoDB write capacity.

---

**User story**
As a system operator, I want ingestion to be decoupled from storage so that write spikes from concurrent agents don't slow down or reject event publication.

---

**Acceptance criteria**
1. An SQS queue and dead-letter queue (DLQ) are provisioned via Terraform
2. The Ingestion API enqueues validated events to SQS — it no longer writes directly to DynamoDB
3. A new Event Processor service consumes messages from SQS and writes events to DynamoDB
4. A message that fails processing is retried up to 3 times before being routed to the DLQ
5. Events are written to DynamoDB with the same schema as before this change
6. The MCP Server requires no changes — the published event format is unchanged

---

**Acceptance tests**
- A valid event published to the Ingestion API appears in DynamoDB after being processed from SQS
- A malformed SQS message that fails processing 3 times is moved to the DLQ
- The Ingestion API returns a 202 response immediately after enqueuing (does not wait for DynamoDB write)

---

**Technical notes**
- Event Processor is a new Node/TypeScript service, triggered by SQS
- DLQ should be provisioned alongside the main queue in the same Terraform module
- No changes to the event schema or DynamoDB table structure
