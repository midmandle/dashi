# 1.2 ŌĆö Publish an event via MCP

Build an MCP server that exposes two tools: `publish_event` and `start_workflow`. Both forward events to the Ingestion API (from 1.1) over HTTP using the shared event schema. `start_workflow` uses a dedicated event type (`workflow_started`) and carries an ordered list of expected event IDs in its payload ŌĆö this manifest is what Dashi uses in Phase 2 to infer progress as subsequent events arrive. `publish_event` is used for all step-level events.

**Expected outcome:** An agent with the MCP server configured can call `start_workflow` to register a workflow and its expected steps, then call `publish_event` for each step ŌĆö Dashi can match each event against the manifest to track progress.

---

**User story**
As an AI agent, I want to be able to call `start_workflow` and `publish_event` MCP tools so that I can declare my workflow structure upfront and record step events without making raw HTTP calls.

---

**Acceptance criteria**
1. An MCP server is available that exposes `publish_event` and `start_workflow` tools
2. `publish_event` accepts the fields defined in the event schema from 1.1 and forwards them to `POST /events`
3. `start_workflow` accepts `workflowId`, `agentId`, `timestamp`, and an ordered list of expected `eventId`s, and publishes a `workflow_started` event to `POST /events`
4. The ordered `eventId` list in `start_workflow` represents the expected steps of the workflow in sequence
5. A valid call to either tool results in the event being persisted via the Ingestion API and returns a success response
6. An invalid call (missing required fields, wrong types) returns a descriptive error to the caller without hitting the Ingestion API
7. The MCP server is configurable with the Ingestion API base URL

---

**Acceptance tests**
- Calling `publish_event` with a valid payload ŌåÆ success response + event persisted via Ingestion API
- Calling `start_workflow` with a valid payload ŌåÆ success response + `workflow_started` event persisted with ordered `eventId` list in `payload.steps`
- Calling either tool with a missing required field ŌåÆ descriptive error, no API call made
- Calling either tool with a wrong field type ŌåÆ descriptive error, no API call made
- Ingestion API returns a non-2xx response ŌåÆ error surfaced to the caller

---

**Technical notes**
- Reuse the Zod schema and inferred TypeScript types from 1.1 ŌĆö the MCP server should depend on the shared schema package, not define its own
- `start_workflow` publishes an event with `type: "workflow_started"`; the steps manifest is carried in `payload.steps` as an ordered array of `eventId` strings
- When events are published via `publish_event`, the `eventId` on each event maps directly to a position in the `payload.steps` list ŌĆö this is how Phase 2 infers progress
- The MCP server should be packaged and runnable via `npx` so it can be referenced directly in an MCP config
