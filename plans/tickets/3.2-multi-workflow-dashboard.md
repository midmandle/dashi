# 3.2 — Multi-workflow dashboard

Introduce a `workflow_completed` event type and update the dashboard to distinguish between in-progress and completed workflows. The default view shows all workflows currently in progress; a separate completed view shows workflows that have finished. Both views are driven by the existing SSE stream.

**Expected outcome:** A user monitoring multiple agents can see all in-progress workflows at a glance, with each showing a live timeline and progress bar. Finished workflows are accessible in a separate completed view.

---

**User story**
As a user monitoring multiple agents, I want to see all in-progress workflows on one screen and review completed ones separately, so I can focus on what still needs attention.

---

**Acceptance criteria**
1. A new `workflow_completed` event type is defined; it carries a `workflowId` linking it to its workflow
2. The default dashboard view shows all workflows that have a `workflow_started` event but no `workflow_completed` event
3. Each in-progress workflow is displayed as a row with its timeline and progress bar
4. When a `workflow_completed` event arrives, the workflow moves from the in-progress view to the completed view without a page refresh
5. A completed view lists all workflows that have received a `workflow_completed` event
6. Completed workflows display their final timeline state and a 100% progress bar
7. Both views update in real time via the existing SSE stream
8. Both views are accessible without a full page reload (tab or toggle on the same page)

---

**E2E test**
1. Publish a `workflow_started` event with 2 steps for workflow A, and a `workflow_started` event with 3 steps for workflow B
2. Open the dashboard — assert both workflows appear in the in-progress view
3. Publish `step_completed` events for all steps in workflow A, then publish a `workflow_completed` event for workflow A
4. Assert workflow A disappears from the in-progress view and appears in the completed view without a page refresh
5. Assert workflow B remains in the in-progress view
6. Navigate to the completed view — assert workflow A is shown with a 100% progress bar

---

**Acceptance tests**
- A workflow with `workflow_started` but no `workflow_completed` appears in the in-progress view
- A workflow with `workflow_completed` does not appear in the in-progress view
- A workflow with `workflow_completed` appears in the completed view with a 100% progress bar
- Receiving a `workflow_completed` event via SSE moves the workflow between views without a page refresh
- Multiple in-progress workflows are all visible simultaneously in the in-progress view

---

**Technical notes**
- `workflow_completed` follows the same event schema as other events: `type: "workflow_completed"`, linked by `workflowId`
- In-progress vs. completed state is determined client-side by the presence of a `workflow_completed` event — no new backend query needed
- The event type enumeration on ticket 1.1's schema should be updated to include `workflow_completed`
- Layout: vertically stacked rows, one per workflow, scrollable
