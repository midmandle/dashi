# 2.2 — Stream processing pipeline

Wire up the backend pipeline from DynamoDB to Redis: new events written to DynamoDB trigger DynamoDB Streams, which invoke a Stream Processor Lambda, which publishes events to Redis. Redis serves as both a pub/sub channel (for the API Server to subscribe to in ticket 2.3) and a recent-event cache (to serve new clients without hitting DynamoDB). All new infrastructure provisioned via Terraform.

**Expected outcome:** A new event written to DynamoDB is automatically published to Redis within seconds, available for pub/sub consumers and readable from the Redis cache.

---

**User story**
As a developer, I want new events to flow automatically from DynamoDB into Redis so that the API Server can deliver them to connected clients in real time.

---

**Acceptance criteria**
1. New events written to DynamoDB are picked up by DynamoDB Streams
2. The Stream Processor Lambda is triggered by DynamoDB Streams and publishes each event to Redis
3. Published events are available on a Redis pub/sub channel
4. Published events are written to a Redis cache (e.g. a list per `workflowId`)
5. All new infrastructure (DynamoDB Streams, Stream Processor Lambda, Redis on ElastiCache) is provisioned via Terraform

---

**Acceptance tests**
- A new event written to DynamoDB appears in the Redis pub/sub channel
- A new event written to DynamoDB appears in the Redis cache
- Multiple events for the same `workflowId` are all present in the Redis cache

---

**Technical notes**
- DynamoDB Streams configured with `NEW_IMAGE` stream view type — only the new event record is needed
- Stream Processor Lambda publishes to a Redis pub/sub channel and appends to a Redis list keyed by `workflowId`
- No direct HTTP call from the Lambda to the API Server — Redis is the only integration point
