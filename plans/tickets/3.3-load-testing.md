# 3.3 — Load testing

Introduce a k6 load testing script that simulates concurrent agents publishing events to the Ingestion API write path. The goal is to establish a performance baseline — recording latency, throughput, and error rates at increasing concurrency levels — so future changes have a benchmark to compare against.

**Expected outcome:** A repeatable, manually-triggered k6 script that graduates through concurrency levels and produces a JSON results file. Baseline results are committed alongside the script.

---

**User story**
As a developer, I want to run a load test against the write path so that I can understand how the system performs under concurrent agent load and have a baseline to compare against future changes.

---

**Acceptance criteria**
1. A k6 script is introduced under `load-tests/` that simulates concurrent agents publishing events to the Ingestion API
2. The script graduates through concurrency levels: 10, 50, 100, 500, and 1000 concurrent agents
3. Events published by the script conform to the real event schema
4. The script reports p50, p95, p99 latency, requests/sec, and error rate per stage
5. Results can be written to a JSON output file via a CLI flag
6. The Ingestion API URL is configurable via an environment variable — no hardcoded endpoints
7. Baseline results from an initial run are committed alongside the script
8. A short README documents how to run the tests and how to read the output

---

**Technical notes**
- Use k6 `stages` to ramp up and hold each concurrency level before stepping to the next
- Each virtual user should simulate a single agent: publish a `workflow_started` event followed by a sequence of `step_started`/`step_completed` pairs
- Run with `k6 run --out json=results/baseline.json load-tests/write-path.js`
- Baseline results file should be committed to `load-tests/results/baseline.json`
