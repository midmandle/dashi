# 1.1 — Accept and store a single event

Define the core event schema as TypeScript types with Zod validation. Build a POST HTTP endpoint that accepts a single event payload, validates it, and writes it to DynamoDB. Provision the Lambda, API Gateway, and DynamoDB table via Terraform. This is the foundational ingestion layer that all other tickets depend on.

**Expected outcome:** A deployed endpoint that agents (or any HTTP client) can POST a valid event to, and which persists it durably in DynamoDB.

---

**User story**
As an AI agent, I want to be able to POST an event to an HTTP endpoint so that my activity is recorded and can be displayed on the dashboard.

---

**Acceptance criteria**
1. A `POST /events` endpoint exists and accepts a JSON body
2. Valid events are written to DynamoDB and a `201 Created` response is returned
3. Invalid events (missing required fields, wrong types) return `400 Bad Request` with a descriptive error
4. The DynamoDB table is provisioned via Terraform
5. The Lambda function and REST API Gateway are provisioned via Terraform
6. The event schema includes at minimum: `eventId`, `workflowId`, `agentId`, `type`, `timestamp`, and an optional `payload`

---

**Acceptance tests**
- `POST /events` with a valid event body → `201` + item written to DynamoDB
- `POST /events` with a missing required field → `400` + descriptive error message
- `POST /events` with a wrong field type → `400` + descriptive error message
- `POST /events` with an empty body → `400`

---

**Technical notes**
- DynamoDB partition key: `workflowId`, sort key: `timestamp#eventId` — `timestamp` drives ordering, `eventId` breaks ties. If callers cannot guarantee `eventId` uniqueness per workflow, fall back to a system-generated UUID as the SK suffix
- Use Zod for runtime validation; export the inferred TypeScript type from the schema
- Lambda behind REST API Gateway, all provisioned via Terraform
