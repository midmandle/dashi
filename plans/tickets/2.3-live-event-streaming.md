# 2.3 — Live event streaming to the dashboard

The API Server (ECS) subscribes to the Redis pub/sub channel from 2.2 and pushes new events to connected browser clients over SSE. The dashboard switches from the one-time fetch in 2.1 to a live SSE connection — existing events are served from the Redis cache on load, and new events arrive in real time without a page refresh. All infrastructure provisioned via Terraform.

**Expected outcome:** An agent publishing an event via MCP is reflected on the dashboard within seconds, with no page refresh required.

---

**User story**
As a user, I want the dashboard to update in real time as agent events arrive so that I can monitor progress without refreshing the page.

---

**Acceptance criteria**
1. The API Server (ECS) subscribes to Redis on startup and pushes all new events to connected clients over SSE
2. On load, the dashboard fetches all existing events from `GET /events`, backed by the Redis cache
3. Subsequent events are delivered to the dashboard via SSE without a page refresh
4. The event list on the dashboard updates in place as new events arrive
5. The API Server on ECS is provisioned via Terraform

---

**E2E test**
1. Open the dashboard in a browser
2. Publish an event via `POST /events`
3. Assert the new event appears on the dashboard without a page refresh

---

**Acceptance tests**
- `GET /events` returns all events from the Redis cache
- Publishing an event via `POST /events` results in it appearing on the dashboard without a page refresh
- A new browser client connecting receives all existing events from the Redis cache on load
- A second browser client connecting while events are being published receives the same events as the first
- Disconnecting and reconnecting the SSE client resumes the stream without duplicating existing events

---

**Technical notes**
- API Server subscribes to Redis on startup; each pub/sub message is forwarded to all active SSE connections
- `GET /events` reads from the Redis cache — no direct DynamoDB reads in this ticket
- SSE stream is global (all events); per-`workflowId` filtering is deferred to future work
- Consider client-side deduplication in case the initial cache fetch and the live SSE stream overlap
