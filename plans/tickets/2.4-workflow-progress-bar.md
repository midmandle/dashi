# 2.4 — Workflow progress bar

Group events by `workflowId` on the dashboard and render a progress bar for each workflow. Progress is inferred client-side: the `workflow_started` event declares the ordered list of expected step `eventId`s, `step_started` indicates a step is in progress, and `step_completed` indicates a step is done. The progress bar reflects the ratio of completed steps to total steps, updating in real time via SSE.

**Expected outcome:** The dashboard displays a progress bar per workflow, updating in real time as new step events arrive.

---

**User story**
As a user, I want to see a progress bar for each workflow so that I can understand how far along each agent is at a glance.

---

**Acceptance criteria**
1. Events are grouped by `workflowId` on the dashboard
2. Each workflow with a `workflow_started` event displays a progress bar
3. Progress is calculated as the number of `step_completed` events whose `eventId` appears in `payload.steps` divided by the total steps declared in `workflow_started`
4. Steps with a `step_started` event but no `step_completed` are shown as in-progress
5. The progress bar updates in real time as new events arrive via SSE
6. A workflow with no `workflow_started` event is handled gracefully (displayed without a progress bar)

---

**E2E test**
1. Publish a `workflow_started` event with 4 step `eventId`s
2. Publish `step_completed` events for 2 of the 4 steps
3. Open the dashboard and assert a progress bar is shown at 50%
4. Publish a `step_started` event for a 3rd step — assert it is shown as in-progress
5. Publish a `step_completed` event for that step — assert the progress bar updates to 75% without a page refresh

---

**Acceptance tests**
- A workflow with 2 of 4 `step_completed` events shows a 50% progress bar
- A workflow with all steps completed shows a 100% progress bar
- A step with `step_started` but no `step_completed` is shown as in-progress
- A workflow with no `workflow_started` event renders without a progress bar
- Progress bar updates in real time as new step events arrive via SSE

---

**Technical notes**
- Progress = count of `step_completed` events with `eventId` in `payload.steps` / `payload.steps.length`
- `step_started` and `step_completed` events reference their step via the top-level `eventId` field
- All progress inference is client-side — no backend changes needed
- Known event types introduced: `workflow_started`, `step_started`, `step_completed` — ticket 1.1's event schema should be updated to enumerate these types on the `type` field
